\section*{Why I Want to Build This}

I debug AI API responses by hand. I am building a research project where a language model calls tools to gather data, and when something breaks --- wrong arguments passed, a reasoning loop that never exits, a tool result the model quietly ignores --- I open an API client to look at the raw response. What I see is a flat \texttt{output[]} array. Reasoning blocks mixed with function calls mixed with results mixed with the final message, all at the same indentation level, no visual hierarchy, no grouping.

I checked Postman, Insomnia, and Thunder Client. All three show the same thing: a JSON tree. None of them treat \texttt{output[]} as structured data. They all treat it as bytes to display.

The Open Responses spec defines exactly what each item type means. The A2UI spec defines how an agent's UI descriptors should be described. The rules exist. The rendering doesn't. This is a clearly bounded gap with a clear solution, and I wanted to build it.

So I spent two weeks reading the API Dash codebase before writing a single line of PoC code. I learned where format detection happens (\texttt{\_resolveViewOptions}), how the genai package abstracts providers, how \texttt{ResponseBodyView} routes to the right widget. Then I built a working prototype --- not to have something to show for this application, but because I was debugging my own AI project and I wanted the tool. I made one architectural mistake: I put A2UI detection in \texttt{ResponseBodySuccess} because that's where widget selection happens. It worked. But that's the wrong layer --- format detection belongs in \texttt{ResponseBody.\_resolveViewOptions()}, where every other format decision already lives. I caught it and moved it before opening the PR. That kind of decision only comes from reading the codebase carefully enough to understand \emph{why} things are structured the way they are.

The PoC is open as \href{https://github.com/foss42/apidash/pull/1358}{PR \#1358}: 1,400+ lines, 9 new files, 39 passing tests, and I use it. GSoC is how I take it to production.

\subsection*{What I Want the Maintainers to Know}

\begin{itemize}[itemsep=8pt]
  \item \textbf{I picked this project because I need the output myself.} Every other GSoC idea I considered was interesting technically. This one I was already building.   \item \textbf{I read the code before I write code.} Every PR I've submitted to API Dash was preceded by reading the relevant parts of the codebase. I don't guess at architecture.

  \item \textbf{I fix my own mistakes.} The A2UI layer placement error above is not the only wrong decision I made during the PoC. It's the one I caught earliest. I expect to make more during GSoC --- the difference is I look for them and correct them before asking for review.

  \item \textbf{I ship across ecosystems.} 11 merged PRs in Python, Java, Go, Rust, and Dart. I can context-switch, read unfamiliar code quickly, and write fixes that fit the project's style. The API Dash genai package is the most complex Flutter package I've worked in, and I understand it well enough to extend it.

  \item \textbf{I am already in the community.} I attend the weekly GSoC calls, I'm active in the \texttt{\#gsoc-foss-apidash} Discord channel, and I participate in the \href{https://github.com/foss42/apidash/discussions/1227}{Idea 5 discussion thread}. I know the other applicants and what they have submitted. My PoC is the only one that includes a working streaming parser and unit tests.
\end{itemize}

\section{Project Proposal Information}

\subsection{1. Proposal Title}

Open Responses Protocol Support \& Generative UI Rendering Engine for API Dash

\subsection{2. Abstract}

I've been testing AI APIs with Postman, Insomnia, and Thunder Client. All three show agentic responses (reasoning + tool calls + results + message) as raw JSON. I have to match \texttt{call\_id} values manually, scroll past reasoning blocks, and figure out the execution flow from a flat array.

I built a working proof-of-concept in API Dash that solves this. It's open as \href{https://github.com/foss42/apidash/pull/1358}{\textbf{PR \#1358}} ($\sim$1,400+ lines, 9 new files, 11 modified, \textbf{39 passing tests}). Video demos: \href{https://youtu.be/paN-KGIhNms}{\textbf{Open Responses Viewer}} $\cdot$ \href{https://youtu.be/T2KbHth736U}{\textbf{A2UI Renderer}}. Idea doc: \href{https://github.com/foss42/apidash/pull/1321}{\textbf{PR \#1321}}.

It adds support for the \href{https://www.openresponses.org/}{Open Responses protocol} and \href{https://github.com/google/A2UI}{A2UI component rendering}:

\begin{itemize}
  \item Auto-detects Open Responses format and switches to a structured card view. Reasoning traces become collapsible cards, tool calls get matched to their results by \texttt{call\_id}, web and file search calls get their own cards, messages render as markdown.
  \item Parses SSE streaming responses via \texttt{OpenResponsesStreamParser} -- the same structured cards appear for both streaming and non-streaming responses.
  \item Renders A2UI component descriptors as real interactive Flutter widgets instead of raw JSON. Detects \texttt{surfaceTitle} and shows it as a header.
  \item \textbf{39 unit tests} (22 for Open Responses model/parser, 17 for A2UIParser) in \texttt{packages/genai}, runnable without Flutter.
\end{itemize}

GSoC adds: live progressive streaming (cards fill in character by character), conversation chaining via \texttt{previous\_response\_id}, DashBot integration, and production-grade test coverage.

You have to mentally track which \texttt{function\_call\_output} matches which \texttt{function\_call} by comparing \texttt{call\_id} values, scroll past reasoning blocks to find the actual message, and reconstruct the execution flow from a flat array.

The \href{https://www.openresponses.org/}{Open Responses protocol} is being adopted across providers -- OpenAI has shipped it. \href{https://github.com/google/A2UI}{Google's A2UI spec} lets agents return UI component descriptors (buttons, forms, tables, charts) instead of just text, and the client renders them as interactive widgets. No tool does this today.

API Dash already handles 60+ response types through one pipeline, has a genai package with provider abstractions, and has SDUI infrastructure. I didn't need to build new plumbing for the PoC.

\textbf{3. A2UI Component Renderer} (\texttt{lib/widgets/a2ui\_renderer.dart} -- \textbf{349 lines})

A2UI sends a JSON list of component descriptors and the client renders UI from that. I built a registry that maps component type strings to Flutter widget builders, and a parser that detects A2UI payloads in JSONL format. I made the renderer stateful so interactive components (checkboxes, text fields) keep their own state. It currently handles \textbf{15 component types}:

\begin{itemize}
  \item \textbf{Layout:} Card, Row, Column, List, Tabs
  \item \textbf{Display:} Text (with h1/h2/h3/caption variants), Image, Icon, Divider

\textbf{4. Response Pipeline Integration}

The existing pipeline has a \texttt{\_resolveViewOptions()} function that picks which viewer to use. I added two new \texttt{ResponseBodyView} options and put the detection logic there:

\begin{sloppypar}
\begin{itemize}
  \item \texttt{structured} view -- kicks in when the response has \texttt{object == "response"} and an \texttt{output[]} array. Routes to the Open Responses viewer.
  \item \texttt{structured + sse + raw} tabs (\texttt{kStructuredSSEBodyViewOptions}) -- kicks in when the SSE output contains \texttt{response.output\_item} events. The \texttt{OpenResponsesStreamParser} reassembles deltas into the same \texttt{OutputItem} list, so the structured viewer renders streaming and non-streaming responses identically.
  \item \texttt{genui} view -- kicks in when the body contains \texttt{createSurface} or \texttt{updateComponents} JSONL keys. Routes to the A2UI renderer.
  \item \texttt{ResponseBodySuccess} then just renders the right one based on which view option was resolved.
\end{itemize}
\end{sloppypar}

\section{How I'll Build This -- Technical Approach for GSoC}

The PoC works on fixture data. Real responses have edge cases I haven't handled yet. Here's what I need to build during GSoC:

\subsection{A. Streaming with Typed SSE Events}

\begin{sloppypar}
Open Responses streaming events are typed and target specific items by index. You get \texttt{response.output\_item.added} when a new item starts, then a series of \texttt{response.output\_text.delta} events that each say ``append this text to item at index N'', and finally \texttt{response.output\_item.done} when that item is complete. So the parser needs to keep a running map of in-progress items and route each delta to the right card in the UI. The PoC viewer already renders each item type as its own card (as shown above), so the streaming layer just needs to feed items into that same widget tree incrementally instead of all at once.
\end{sloppypar}

\subsection{B. Conversation Chaining (\texttt{previous\_response\_id})}

Open Responses supports continuing a conversation by passing the previous response's ID in the next request. Right now API Dash treats every request as standalone, so I need to change that:

\begin{itemize}
  \item When we parse a response, pull out the \texttt{response.id} and store it on the request model.
  \item Add a field to the request editor so you can paste a \texttt{previous\_response\_id} to chain off a prior response.
  \item Build a timeline view that links chained requests so you can see the whole conversation flow in one place.
  \item Each turn in the chain should be expandable/collapsible so you're not overwhelmed if the conversation is long.
\end{itemize}

\subsection{C. Extended A2UI Component Support}

I have 15 component types in the PoC. That's enough for a demo, not enough for real use. During GSoC I need to cover enough of the A2UI spec that real Vertex AI agent responses just work:

\begin{itemize}
  \item \textbf{More input components:} DatePicker, Slider, Radio, Dropdown (these are common in agent UIs but not in the PoC yet)
  \item \textbf{Data display:} Table with sorting and filtering, Charts via \texttt{fl\_chart}
  \item \textbf{Navigation components:} Accordion, Stepper -- needed for multi-step agent flows
  \item \textbf{Form state:} Right now state is per-component; I need to unify it so a form submission can read all field values at once
  \item \textbf{Streaming:} A2UI uses a flat ID-referenced model where you can add/update components incrementally. Components should render as they arrive, not wait for the whole payload
\end{itemize}

\subsection{D. DashBot Integration}

DashBot passes everything through \texttt{MarkdownBody} inside \texttt{ChatBubble}. Works for plain text, but Open Responses format shows up as raw JSON. Fix: check the response format before rendering:

\begin{itemize}
  \item If it's Open Responses format, run it through the same \texttt{OpenResponsesResult.fromJson} pipeline and render the structured cards inline in the chat (reasoning collapsed by default, tool calls as expandable cards).
  \item If it contains A2UI components, render the interactive widgets right inside the chat bubble, so the agent's UI shows up where the conversation is.
  \item Plain text/markdown responses stay exactly as they are now.
\end{itemize}

\subsection{What the End Result Will Look Like}

\textbf{Scenario 1: Non-streaming Open Responses.} You send a request. API Dash detects the format and switches to structured view. Reasoning card collapsed at top, tool call card with matched result below it, message as markdown at bottom. Token usage bar at the very end.

\textbf{Scenario 2: Streaming.} Cards appear live as SSE events arrive. Reasoning card fills in progressively, then a function call card shows up with arguments growing character by character, then the result slides in, then the message. No raw JSON fragments visible at any point.

\textbf{Scenario 3: A2UI agent response.} A Vertex AI agent returns component descriptors. Instead of JSON, you see rendered Flutter widgets: buttons you can tap, text fields you can type in, data tables with rows. You test the agent's UI without building a separate frontend.

\textbf{Scenario 4: Multi-turn debugging.} Three chained requests using \texttt{previous\_response\_id}. API Dash links them into a timeline. You see the full flow in one connected view instead of three separate tabs.

\section{Why Me}

I am the person who already built the PoC. Not to have something to show for an application --- because I was debugging my own AI project and I needed the tool. Here is what that actually required:

I started with \href{https://github.com/foss42/apidash/pull/1279}{PR \#1279} (input validation) to understand how Riverpod state flows through the app before touching anything deeper. Then \href{https://github.com/foss42/apidash/pull/1290}{PR \#1290} (custom LLM providers), which required reading the entire genai package: how \texttt{ModelProvider} abstracts backends, how \texttt{createRequest()} formats input, how \texttt{outputFormatter} pipes results back. I know that pipeline well enough to extend it, not just call it. Then I built the PoC: sealed class hierarchy for the Open Responses output types, 7-card structured viewer, A2UI renderer with 15 components, SSE streaming parser, 39 unit tests. All before applying. \href{https://github.com/foss42/apidash/pull/1358}{PR \#1358} is the result.

\textbf{What the PoC already proves:}
\begin{itemize}
  \item The format auto-detection works: Open Responses JSON, Open Responses SSE stream, and A2UI JSONL each route to the right viewer with zero false positives on regular JSON responses.
  \item The sealed class dispatch is correct: all five output item types (\texttt{Message}, \texttt{Reasoning}, \texttt{FunctionCall}, \texttt{WebSearchCall}, \texttt{FileSearchCall}) each render as their own card type.
  \item The A2UI renderer handles interactive state: TextFields, Checkboxes, and Switches keep local state; unknown component types fall back gracefully without crashing.
  \item The streaming parser assembles a complete \texttt{OutputItem} list from SSE deltas --- the same viewer renders streaming and non-streaming responses identically.
  \item The models live in \texttt{packages/genai} and are tested without Flutter. The test suite runs in 39ms.
\end{itemize}

The remaining GSoC work (progressive streaming, conversation chaining, DashBot integration, extended A2UI components) is extension, not invention. The foundation is already in the codebase and passing tests.

\textbf{What will be hard:}

\begin{itemize}
  \item \textbf{Streaming (live progressive rendering)} -- The PoC has \texttt{OpenResponsesStreamParser} which reassembles SSE deltas into a final \texttt{OutputItem} list and the structured viewer already renders it correctly. What's missing for GSoC is the \textit{progressive} part -- cards should fill in character by character as deltas arrive, not wait for the stream to complete. This means wiring the parser's incremental state into a Riverpod stream so each delta triggers a widget rebuild at the right card, not a full re-render.
